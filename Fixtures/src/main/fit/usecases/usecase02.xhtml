Use Case 02 : User orders lunch

Main Scenario

  1. 1 User logs in
  2. System informs user of the current balance and shows available products
  3. 3 User selects lunch by adding available products to order
  4. 4 System verifies the remaining balance and informs user of balance and order
  5. repeat 3-4 till order complete
  6. 5 User confirms choice
  7. 6 System validates and updates balance, registers order
  8. 7 System confirms order and wishes the User a nice day

Alternatives

  • 2a Remaining balance is too low to order any product

    1. System informs user that nu order is possible
    2. User adds money to Balance

  • 2b Remaining balance is too low to offer all products

    1. - 2b1 The System shows the products but makes them unavailable

  • 6a User tries to cheat and confirms multiple orders of which the sum exceeds the balance.

    rationale :

    The user opens several sessions and builds several orders for which the total exceeds the balance. It depends strongly on the implementation but is a risk worth checking for. It serves to verify the validation is against the persistent record f the balance, not some cached value in a session or conversation.

    1. Validation of remaining balance fails
    2. System informs user the remaining balance is insufficient.
    3. continue from 2
